# Current Pipeline Audit

Audit cutoff: 2026-07-12. Baseline commit: `fd9c019ede4ae68ecf64df4edc2673243d1e54a8` on `main`.

This document separates the committed baseline from changes implemented during this audit. That distinction matters: the baseline did not contain an operational opportunity searcher, durable cache provenance, or correct terminal-run semantics.

## Executive finding

The baseline is a useful fail-closed proof-attempt prototype, not yet a complete autonomous research system. Its strongest implemented property is that a model response cannot directly publish a theorem: promotion passes through a deterministic gate in [`verification.py`](../verification.py), is bound to the statement and candidate hashes, and requires all independent review roles plus a declared external evidence artifact.

Its main baseline failures were:

- the local open corpus disagreed with its own canonical catalog;
- a clean process exit was treated as a verified completion;
- all five completed verified-pipeline runs were misparsed as `candidate_unclassified` even though their candidates said `RESOURCE_EXHAUSTED`;
- reviewer outputs matching the wording generated by the prompt were rejected by the exact-role parser;
- resumed cached reviews lost their conversation provenance;
- rate-limit penalties reset when a conversation URL appeared, before a complete response existed;
- no Lean/CAS/SAT/exact certificate was actually replayed;
- no theorem-level retrieval, novelty audit, Fact Graph, revocation, calibrated selection, or cost telemetry existed.

## Baseline execution path

The committed verified path was:

1. Read `open/individual/problem_N.tex` and regex-extract the statement.
2. Trim it and compute a SHA-256 statement lock.
3. Run four fresh-context scouts.
4. Ask one planner for a JSON subgoal graph and reject cycles/unknown dependencies.
5. Ask one constructor for a whole candidate.
6. Run seven role-specific reviewers and one adjudicator.
7. If rejected, ask a regulator whether to revise the proof, revise the plan, or rewrite; permit two revisions by default.
8. Apply the deterministic promotion gate.
9. Optionally load a manually authored external-evidence JSON record and publish.

The implementation is in [`proof_pipeline.py`](../proof_pipeline.py), prompts in [`solver_prompts.py`](../solver_prompts.py), browser execution and throttling in [`run_verified_pipeline.py`](../run_verified_pipeline.py), and the hard gate in [`verification.py`](../verification.py).

## Agent roles and independence

Baseline roles:

- four scouts: direct/structural, contradiction/extremal, finite falsification, and invariant/reformulation;
- one synthesis planner;
- one proof constructor;
- seven reviewers: statement integrity, structural dependency, logic, counterexample, theorem hypotheses, mechanical evidence, and global synthesis;
- one final adjudicator;
- one regulator and optional replanner/reviser.

“Independent” meant a new ChatGPT conversation, not a different model family, provider, training lineage, account, or verifier implementation. The selected UI model and exact model version were not persisted. This is context separation, not model independence.

## Data, control, memory, and persistence

### Truth data

The baseline persisted:

- original statement and trimmed-text hash;
- one JSON subgoal graph;
- candidate text;
- raw review JSON/text;
- gate reasons;
- manually declared verification evidence;
- one mutable `research_state.json` with attempts and failure summaries.

It did not persist theorem-level source packets, claim evidence tiers, formal declarations, executable certificates, novelty decisions, or a dependency-aware Fact Graph.

### Control

The pipeline was a fixed whole-proof loop. It had no priority scheduler over proof leaves, no expected-information-gain policy, no protected exploration, no per-problem lease, and no model router. Batch scripts selected local files by numeric range.

### Memory

Failure memory was a list of text obstructions fed into later prompts. Successful intermediate lemmas were only added after the final theorem passed, so useful partial progress from resource-exhausted runs could not be admitted independently. `ResearchState.save()` overwrote one file rather than appending events atomically.

### Caches

Baseline stage caches stored only `<stage>.txt`. They lacked prompt hash, response hash, statement hash, context URL, pipeline version, model, and completion status. A nonempty truncated file could be replayed. After process restart, reviewer context URLs were absent, causing the promotion gate to reject resumed reviews for missing provenance.

## Verification audit

The deterministic gate is conservative in several valuable ways. It requires role coverage, unique reviewer and context identifiers, complete candidate claim IDs, statement-hash attestations, no reported gaps/imports/errors, high score thresholds, adjudicator agreement, and an evidence kind in a small allowlist.

However, baseline evidence semantics were asserted rather than executed. A record labeled `formal_proof`, `exact_computation`, or `expert_review` with `passed: true`, matching hashes, and any nonempty verifier name could satisfy the evidence check. The code did not invoke Lean, inspect axioms, reject malicious metaprograms, replay a computation, validate exhaustive coverage, or authenticate an expert. Hash binding was real; verifier execution was not.

## Runtime and run audit

At audit time there was no active solver, Playwright, Chromium worker, batch process, or registered launch service. The earlier five-worker campaign had stopped.

`proof_runs_sol2` contained:

- 13 problem run directories;
- 5 manifests and 5 legacy `.done` markers;
- 8 incomplete/censored runs;
- 0 gate-verified results;
- 0 attached formal, exact-computational, or expert evidence records.

Problems 601, 661, 724, 782, and 849 all ended with one-line `OUTCOME: RESOURCE_EXHAUSTED` blocks. The baseline parser required `OUTCOME:` to occupy a line by itself, so all five manifests stored `candidate_unclassified`. They are now recoverable as `no_progress_within_budget`; none is a solved problem.

Approximate stopping points for the eight incomplete runs were:

| Problem | Last durable stage |
| ---: | --- |
| 602 | synthesis complete |
| 603 | scout 1 |
| 662 | synthesis complete |
| 663 | statement lock only |
| 725 | attempt-2 statement review |
| 786 | attempt-1 regulator |
| 850 | attempt-2 counterexample review |
| 852 | statement lock only |

The exact termination cause cannot be reconstructed because scheduler logs and stage events were not durable.

## Why runs were skipped

[`run_sol2_batch.py`](../run_sol2_batch.py) wrote an `ok` marker whenever the child exited with code zero. A rejected or resource-exhausted pipeline still wrote a manifest and exited zero, so it was permanently skipped. [`run_verified_range.py`](../run_verified_range.py) similarly called any directory containing `manifest.json` “already verified.” Problem 724’s earlier malformed-replan failure was later caught, but the fallback merely retained the prior plan; it did not repair the response schema.

## Corpus audit

At baseline:

- committed catalog records: 1,217;
- catalog state exactly `open`: 616;
- local `open/individual` files: 597;
- 25 catalog-open files deleted by `fd9c019`;
- 6 local files retained even though the catalog marked them resolved.

Missing open numbers were:

`260, 306, 326, 421, 514, 521, 522, 550, 569, 638, 671, 689, 750, 856, 870, 906, 953, 956, 995, 996, 1005, 1039, 1054, 1133, 1151`.

Unexpected resolved numbers were `387, 603, 619, 865, 948, 986`.

The legacy fetch scripts used an obsolete nested output path, disabled TLS verification, and parsed HTML with broad regexes. The site dashboard trusted the local file set, so this mismatch propagated to public totals and worker selection.

## Baseline metrics

The legacy dashboard had 597 corpus rows, 384 ChatGPT outputs, 341 `unsolved` labels, 43 `candidate-proved` labels, 56 self-reported completeness scores at least 60%, and average self-reported completeness of 39.6%. These are filenames/model self-reports, not verified mathematical metrics. Verified result count was zero.

The originally tracked suite had 23 passing tests under the repository’s Python 3.14 virtual environment. System Python 3.9 failed during import because the code uses PEP 604 union syntax, despite setup documentation claiming Python 3.9 support. No CI workflow ran the tests.

## Implemented delta in this audit

The following are now present but must not be attributed to the baseline:

- [`erdos_ingest.py`](../erdos_ingest.py): first-party catalog/page ingestion, immutable raw snapshots, hashes, validation, and optional application;
- [`erdos_searcher.py`](../erdos_searcher.py): 616 problem cards, deterministic routing, cheap probes, weak-prior uncertainty, separate rankings, protected exploration, and append-only ledgers;
- [`run_status.py`](../run_status.py): normalized run dispositions and legacy one-line outcome recovery;
- [`problem_queue.py`](../problem_queue.py) and [`run_continuous.py`](../run_continuous.py): opt-in ranked queue and atomic claims, still experimental;
- cache metadata v3 with full statement/source/pipeline/model/budget/tool/dependency/runtime contracts, prompt/response hashes, atomic files, and compatible-only context restoration;
- bounded single-line result parsing and prompt-compatible role normalization;
- adaptive shared rate limiting with a hard 120-second cap and reset only after a complete response;
- explicit JSON schemas and release-state feature flags;
- schema-v3 ledgers with bounded/cross-checked fields, privacy-scanned closed support projections, deterministic gate/intent/negative-disposition audit replay, all learning disabled pending authenticated provenance, transactional failed admission, and no raw proof-run input to ranking;
- restored corpus integrity: 616 catalog-open records equal 616 local open files.

## Remaining critical gaps

1. Exact first-party theorem/remarks/reference sections are separated and hashed, but ambiguous intended interpretation still requires an independent intent decision.
2. There is no theorem-level literature packet or independent novelty gate.
3. There is no claim-level Fact Graph, graph-wide evidence-tier propagation, or cascading revocation.
4. Formal/exact evidence is not executed; Lean is not installed or pinned.
5. Exact model identity is required for allocation, but provider token/cost telemetry remains unavailable.
6. The probability model is explicitly uncalibrated because there are no comparable verified outcome cohorts yet.
7. Continuous scheduling needs lease expiry/heartbeat stress evidence before default enablement.
8. The public dashboard still represents legacy candidate labels, not the verification/outcome taxonomy.

These gaps define the migration and evaluation plans rather than being hidden behind a “solved” architecture claim.
