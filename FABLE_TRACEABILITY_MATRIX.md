# FABLE Requirement Traceability Matrix

## Audit rules and status accounting

This matrix is keyed only by the independently extracted `F-REQ-*` inventory. It contains exactly one record for each `F-REQ-001` through `F-REQ-216`; the 18 §13.6 acceptance criteria remain separate records (`F-REQ-174` through `F-REQ-191`). Prior-ledger references are comparison aids, not the controlling requirement definition.

`VERIFIED` is used only where faithful local semantics, a production CLI call path, positive and negative tests, runtime evidence, and fail-closed behavior were all demonstrated. A library helper, Protocol, registry, mock, fake fetcher/checker, or passing shape test is never enough. `BLOCKED-EXTERNAL` is used only where the remaining decisive step requires unavailable infrastructure; repository-local gaps remain `PARTIAL`, `UNREACHABLE`, `TEST-ONLY`, `UNVERIFIED`, or `OMITTED`.

Runtime evidence below is focused evidence observed during remediation, not a substitute for the fresh aggregate run recorded separately in `FABLE_FINAL_VERIFICATION.md`. “Production” means the strengthened `python -m egmra.cli run` slice unless a row expressly names the legacy promoter. The existing Erdős scheduler does not launch EGMRA, and no API/service daemon exists.
